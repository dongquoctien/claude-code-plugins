# NgRx feature store — derived from real anchor features

This file points at **`.claude/angular-admin-design/reference-features.json`** as the source of truth for every pattern below. Copy from the anchor features, do not invent.

Anchors (default for oh-admin):
- `bs-event` — full CRUD with grid + multiple lookup actions + popups (canonical)
- `ho-reservation-list-ota` — read-heavy search/grid feature
- `ad-dashboard` — chart/dashboard with cross-feature data

If a project uses different anchors (configured in `config.anchorFeatures`), regenerate `reference-features.json` and replace the snippets in this doc with the new project's code.

---

## Mandatory folder layout

Every feature module generated by /aad-generate MUST have:

```
routes/<feature>/
├── <feature>.module.ts            // NgModule + StoreModule.forFeature + EffectsModule.forFeature
├── <feature>-routing.module.ts    // single-child route + data.{title, store}
├── actions/
│   ├── <feature>.actions.ts       // createAction declarations
│   └── index.ts                   // `import * as ThisActions from './...'; export { ThisActions };`
├── reducers/
│   ├── <feature>.reducer.ts       // INNER featureKey + State + adapter + initialState + reducer fn + local selectors
│   └── index.ts                   // OUTER featureKey + ModuleState + combineReducers + top-level selectors
├── effects/
│   ├── <feature>.effects.ts       // class with createEffect blocks + _catchErr helper
│   └── index.ts                   // `export * from './<feature>.effects';`
├── services/
│   ├── <feature>-api.service.ts   // ApiService wrapper with succeedYn validation
│   └── index.ts
├── models/
│   ├── <feature>.model.ts         // I-interfaces (request, response, view-model)
│   ├── api-path.models.ts         // PLURAL: api-path.models.ts not api-path.model.ts
│   └── index.ts
├── containers/
│   ├── <feature>.component.{ts,html,scss}
│   └── index.ts
└── components/
    ├── <feature>-search.component.{ts,html,scss}
    ├── <feature>-form.component.{ts,html,scss}
    └── index.ts
```

**Naming conventions (verified against bs-event):**
- `models/api-path.models.ts` — **plural** (bs-event uses this; v0.1.0 generated singular `api-path.model.ts` which is inconsistent).
- Style extension is **`.scss`**, NOT `.css`. Check `catalog.shared[*].componentPath` siblings to confirm per project.
- Section sub-components: `<feature>-<section>.component.ts` — TÁCH ra, container chỉ orchestrate. Do NOT inline a 200-line container with search + grid + toolbar.

## reducers/<feature>.reducer.ts — sub-reducer

Inner featureKey suffix matches the slice's purpose: `List`, `Main`, `Form`, `Detail`. Pick `List` if there's a grid; use the bs-event naming.

```typescript
import { on, createReducer } from '@ngrx/store';
import { createEntityAdapter, EntityAdapter, EntityState } from '@ngrx/entity';
import * as _ from 'lodash';

import { IMasterSearchResultItem } from '../models';
import { ThisActions } from '../actions';

export const featureKey = 'bsEventList';  // ← INNER key

export interface State extends EntityState<IMasterSearchResultItem> {
  // 기초코드 (lookup codes)
  stationTypeCodeList: any[];
  itemCategoryCodeList: any[];

  // 검색관련 (search)
  loading: boolean;
  searchCondition: any;

  // 그리드관련 (grid state)
  gridDataTotal: number;
  gridState: any;            // ← see knowledge/kendo-grid-patterns.md for the shape
  gridCheckedState: any;

  // 팝업관련 (popups)
  isOpenDetailPopup: boolean;
  selectedDetailRowInfo: IMasterSearchResultItem;
}

export const adapter: EntityAdapter<IMasterSearchResultItem> = createEntityAdapter<IMasterSearchResultItem>({
  selectId: (entityItem: IMasterSearchResultItem) => entityItem.rowid,
  sortComparer: false,
});

const initialState: State = adapter.getInitialState({
  stationTypeCodeList: null,
  itemCategoryCodeList: null,
  loading: false,
  searchCondition: null,
  gridDataTotal: null,
  gridState: {
    pageInfo: { skip: 0, take: 20 },   // ← REAL gridState shape — see kendo-grid-patterns.md
    sortInfo: [],
  },
  gridCheckedState: null,
  isOpenDetailPopup: false,
  selectedDetailRowInfo: null,
});

export const reducer = createReducer(
  initialState,
  on(ThisActions.loadList, (state) => ({ ...state, loading: true })),
  on(ThisActions.getListSuccess, (state, { dataList, totalCount }) =>
    adapter.setAll(dataList, { ...state, loading: false, gridDataTotal: totalCount }),
  ),
  on(ThisActions.getListFailure, (state) => ({ ...state, loading: false })),
);

// local selectors — used by reducers/index.ts top-level selectors
export const selectLoading = (state: State) => state.loading;
export const selectGridState = (state: State) => state.gridState;
export const selectGridDataTotal = (state: State) => state.gridDataTotal;
export const { selectAll: selectAllItems } = adapter.getSelectors();
```

**Critical: `gridState` shape is `{ pageInfo: {skip, take}, sortInfo: [] }`** — NOT flat `{skip, take, sort, filter}`. Verified against `bs-event/reducers/bs-event.reducer.ts:58-61`. v0.1.0 generated flat shape which doesn't match the `IBasisGridState` interface — caused TypeScript errors at the container.

## reducers/index.ts — outer wrapper

```typescript
import { createSelector, createFeatureSelector, combineReducers, Action } from '@ngrx/store';
import * as _ from 'lodash';

import * as fromList from './bs-event.reducer';
import * as fromRoot from 'src/app/reducers';

export const featureKey = 'bsEvent';  // ← OUTER key

export interface BsEventState {
  [fromList.featureKey]: fromList.State;
}

export interface State extends fromRoot.State {
  [featureKey]: BsEventState;
}

export function reducers(state: BsEventState | undefined, action: Action) {
  return combineReducers({
    [fromList.featureKey]: fromList.reducer,
  })(state, action);
}

// top-level selectors (used by container + effects)
export const selectModuleState = createFeatureSelector<State, BsEventState>(featureKey);

export const selectListState = createSelector(selectModuleState, (state: BsEventState) => state[fromList.featureKey]);

export const selectLoading = createSelector(selectListState, fromList.selectLoading);
export const selectGridState = createSelector(selectListState, fromList.selectGridState);
export const selectGridDataTotal = createSelector(selectListState, fromList.selectGridDataTotal);
export const selectAllItems = createSelector(selectListState, fromList.selectAllItems);
```

**Two-level featureKey is non-negotiable.** Pattern (`bs-event`):
- Outer key = camelCase feature name (`bsEvent`) for the module slice
- Inner key = outer + suffix (`bsEventList`) for the sub-reducer

The wrapper exists so future sub-reducers (e.g. `bsEventForm`, `bsEventDetail`) can be added without changing every selector. Skipping this wrapper and flattening to one reducer breaks consistency with the rest of the codebase.

Top-level selector should be named **`selectModuleState`** (matches `bs-event/reducers/index.ts:24`). v0.1.0 generated `selectFeatureState` — inconsistent.

## effects/<feature>.effects.ts — canonical from bs-event

```typescript
import { Injectable } from '@angular/core';
import { Observable, of } from 'rxjs';
import { catchError, concatMap, map, switchMap, tap, withLatestFrom } from 'rxjs/operators';
import * as _ from 'lodash';
// ngrx
import { Store, select } from '@ngrx/store';
import { Actions, createEffect, ofType } from '@ngrx/effects';
import { ThisActions } from '../actions';
import * as fromModule from '../reducers';
// services
import { BsEventApiService } from '../services';
import { ErrorHandlerService } from 'src/app/core/services/error/error-handler.service';
import { ConfirmationService, Confirmation } from 'src/app/shared/modules/alert/alert.service';
// models
import { IMasterSearchResultItem } from '../models';
import { TypedAction } from '@ngrx/store/src/models';

@Injectable()
export class BsEventEffects {
  // Cached selector observables (used inside withLatestFrom)
  stationTypeCodeList$: Observable<any>;
  itemCategoryCodeList$: Observable<any>;

  constructor(
    private actions$: Actions,
    private service: BsEventApiService,
    private _errorService: ErrorHandlerService,
    private alertService: ConfirmationService,
    private store: Store<fromModule.State>,
  ) {
    this.stationTypeCodeList$ = store.pipe(select(fromModule.selectStationTypeCodeList));
    this.itemCategoryCodeList$ = store.pipe(select(fromModule.selectItemCategoryCodeList));
  }

  // ─── Grid list effect (representative pattern) ───────────────────────
  loadList$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ThisActions.loadList),
      switchMap(({ condition }) =>
        this.service.getList(condition).pipe(
          map(({ dataList, totalCount }) => ThisActions.getListSuccess({ dataList, totalCount })),
          catchError((error) => {
            const msg = 'List lookup failed. Please try again in a moment.';
            const actions = [ThisActions.getListFailure({ error: msg })];
            this._catchErr(error, msg, actions);
            return [];
          }),
        ),
      ),
    ),
  );

  // ─── Conditional cache-aware lookup effect ──────────────────────────
  // Pattern: if cached list already exists in store, dispatch *Success with cached
  // data; otherwise call API. Used for lookup/code lists that rarely change.
  loadStationTypeCodeList$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ThisActions.loadStateTypeCodeList),
      concatMap((action) => of(action).pipe(withLatestFrom(this.stationTypeCodeList$))),
      switchMap(([c, stationTypeCodeList]) => {
        if (_.isNil(stationTypeCodeList) || stationTypeCodeList.length === 0) {
          const condition = { masterCode: 'BS002', referCode1: 'EVENT' };
          return this.service.getDetailCode(condition).pipe(
            map((dataList) => ThisActions.getStateTypeCodeListSuccess({ dataList })),
            catchError((error) => {
              this._catchErr(error, 'List lookup failed.');
              return [];
            }),
          );
        } else {
          return of(ThisActions.getStateTypeCodeListSuccess({ dataList: stationTypeCodeList }));
        }
      }),
    ),
  );

  // ─── _catchErr helper — defined private at the bottom ────────────────
  private _catchErr(error: any, customMessage: string, actions?: TypedAction<any>[]) {
    const confirmation = this._errorService.errorHandler(error, customMessage);
    if (_.isNil(confirmation)) {
      return;
    }
    if (_.isNil(confirmation.accept)) {
      if (actions?.length > 0) {
        confirmation.accept = () => {
          actions.forEach((a) => this.store.dispatch(a));
        };
      } else {
        confirmation.accept = () => {};
      }
    }
    this.alertService.alert(confirmation);
  }
}
```

**Critical fixes from v0.1.0:**

1. **Error handling method is `errorHandler` (not `handleError`).** Verify in catalog: `coreServices.find(s => s.className === 'ErrorHandlerService').methods` shows `apiErrorHandler`, `errorHandler`, `parseApiError`, `invokeConsoleMethod`. There is NO `handleError` method.

2. **The `_catchErr` private helper is the project's convention.** Every effects file wraps `this._errorService.errorHandler(...)` in a helper that:
   - Receives `(error, customMessage, actions?)`
   - Calls `errorService.errorHandler(error, customMessage)` → returns `{ message, accept }`
   - Wraps `accept` to dispatch failure actions when user clicks confirm
   - Calls `alertService.alert(confirmation)`

3. **Effects depend on `ConfirmationService` from `shared/modules/alert/alert.service`.** Generated module must import this — verify in catalog.shared.

4. **`return [];` after catchError** (not `return of(...)`) — this terminates the action stream without dispatching anything extra; the `_catchErr` helper has already dispatched the failure action.

5. **`concatMap + withLatestFrom` for cache-aware effects** is the project's lookup pattern. Don't invent a different approach for "load if not loaded" effects.

## services/<feature>-api.service.ts — canonical from bs-event

```typescript
import { Injectable } from '@angular/core';
import { Observable, of, throwError } from 'rxjs';
import { switchMap, catchError } from 'rxjs/operators';
import * as _ from 'lodash';

// services
import { ApiService } from 'src/app/core/services/api/api.service';

// models
import { BsEventApiPath as ApiPath } from '../models';

@Injectable({
  providedIn: 'root',
})
export class BsEventApiService {
  constructor(private api: ApiService) {}

  // ─── Detail-code lookup (BS group code) ──────────────────────────────
  getDetailCode(condition: any): Observable<any[]> {
    const params = _.assign({}, this.api.defaultParams, { condition });
    return this.api.post(ApiPath.postCommonDetail, params).pipe(
      switchMap((res) => {
        if (res.succeedYn && res.result) {
          return of(res.result.list);
        } else {
          return throwError(res || 'api response error');
        }
      }),
    );
  }

  // ─── Grid list with rowid injection ──────────────────────────────────
  getList(condition: any): Observable<any> {
    const params = _.assign({}, this.api.defaultParams, { condition });
    return this.api.post(ApiPath.postEvent, params).pipe(
      switchMap((res) => {
        if (res.succeedYn && res.result) {
          const { list, totalCount } = res.result;
          const dataList = this._addRowid(list);
          return of({ dataList, totalCount });
        } else {
          return throwError(res || 'api response error');
        }
      }),
    );
  }

  // _addRowid pattern — adds a stable rowid for grid use when backend uses
  // a different ID field. Required for adapter.selectId in the reducer.
  private _addRowid(list: any[]): any[] {
    return (list || []).map((item, idx) => ({ ...item, rowid: item.rowid || idx + 1 }));
  }
}
```

**Critical fixes from v0.1.0:**

1. **API path enum is `<Feature>ApiPath` imported from `models/api-path.models.ts`** (plural). bs-event uses `import { BsEventApiPath as ApiPath } from '../models';`.

2. **`api.defaultParams` is a GETTER, not a field.** It returns `{ language, country, stationTypeCode, currency, condition }`. Use `_.assign({}, this.api.defaultParams, { condition })` to spread + override. v0.1.0 used `_.assign({}, this.api.defaultParams, { condition })` correctly — keep this.

3. **`res.succeedYn && res.result` validation is the br-05 envelope.** Failure throws — effect catches.

4. **`api.post` for query-style endpoints, `api.get` for read-only path params.** The bs-event project uses POST for lookups (carrying condition as body). Check `catalog.coreServices['ApiService'].methods` — only what's in there is callable. v0.1.0 used `api.get` blindly even when bs-event project conventions use POST.

5. **`api.postBlob` for Excel/file download** — verify against catalog. bs-event uses it in `getOrderListExcel` etc.

**Critical:**
- `CONTAINERS` and `COMPONENTS` are exported as **module-level constants** with the array form `[Component1, Component2]`. The `@NgModule` declarations field uses **the constants directly**: `declarations: [COMPONENTS, CONTAINERS]` (NOT `[...COMPONENTS, ...CONTAINERS]`).

- Module imports order: Angular built-ins → Kendo → custom shared → routing → NgRx feature.

## What NOT to do (lessons from v0.1.0)

- ❌ Generate `this._errorService.handleError(error)` — method does not exist. Use `_catchErr(error, msg)` helper, which calls `errorHandler`.
- ❌ Import `from 'file-saver'` without checking `catalog.dependencies.installed`. oh-admin does NOT install file-saver — use `URL.createObjectURL(blob) + <a download>` instead.
- ❌ Generate `gridState: { skip: 0, take: 50, sort: [], filter: {} }` — that's flat shape. Correct is `{ pageInfo: { skip, take }, sortInfo: [] }`.
- ❌ Generate `IColumnOption[]` with `width: '200px'` (string) or `template: 'tplName'` (string). Width is `number`, template is `TemplateRef<any>`.
- ❌ Skip the `useTitleAttrYn: boolean` field — it's required (no `?`) in `IColumnOption`.
- ❌ Generate `selectFeatureState` for outer top-level selector — bs-event uses `selectModuleState`. Match.
- ❌ Use `.css` for component styles when the project uses `.scss`.
- ❌ Generate `models/api-path.model.ts` (singular) — project uses plural `models/api-path.models.ts`.

Every "what NOT to do" maps to a v0.1.0 bug. The code-generator must consult the catalog OR `reference-features.json` before each decision.
